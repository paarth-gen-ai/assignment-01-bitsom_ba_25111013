## Storage Systems

This healthcare analytics platform employs a multi-database storage strategy, with each system selected for its specific strengths in handling particular data types and access patterns. PostgreSQL serves as the primary relational database for treatment history, patient demographics, medication records, and staff inputs. The choice of PostgreSQL is driven by its strict ACID compliance, which is non-negotiable in healthcare where data integrity directly impacts patient safety. Features like row-level locking, transaction isolation, and foreign key constraints ensure that concurrent updates from multiple clinicians never corrupt patient records. Additionally, PostgreSQL's JSONB column type allows hybrid storage of semi-structured clinical notes alongside rigidly structured fields, providing flexibility without sacrificing relational integrity.

The Vector Database (such as Pinecone or Milvus) powers the semantic search capability that enables clinicians to query the system in plain natural language. When a doctor asks "Show me patients with similar responses to drug X", the vector DB converts the query into embeddings and retrieves semantically related records from millions of clinical notes that keyword search would never surface. This is particularly critical in a hospital setting where medical terminology varies across physicians, abbreviations are common, and the same clinical concept can be expressed in numerous ways. The vector layer transforms unstructured clinical documentation into a queryable knowledge base.

The Data Warehouse (Snowflake) functions as the central repository for all aggregated, cleaned, and transformed data used in monthly and quarterly reporting. It consolidates information from PostgreSQL, the time-series database, and the vector database into a unified schema optimized for analytical queries. This separation ensures that heavy reporting workloads never interfere with the operational performance of the transactional systems. BI tools like Tableau connect directly to the warehouse for executive dashboards tracking readmission rates, average length of stay, ICU utilization, and physician productivity metrics.

Finally, the Time-Series Database (InfluxDB) handles the high-frequency ICU vital signs stream, which can generate thousands of data points per patient per hour. Traditional relational databases would choke on this volume and velocity. InfluxDB provides native support for time-based queries, automatic data retention policies, and efficient downsampling, making it ideal for monitoring continuous physiological signals like heart rate, blood pressure, oxygen saturation, and temperature.

## OLTP vs OLAP Boundary

The boundary between the Online Transaction Processing (OLTP) system and the Online Analytical Processing (OLAP) system in this architecture is defined at the ETL layer. The OLTP side encompasses PostgreSQL for patient records and InfluxDB for vital signs, handling all real-time clinical operations: admitting patients, recording vitals, prescribing medications, and documenting clinical notes. These systems prioritize low-latency writes, transactional consistency, and concurrent read-write operations from hundreds of bedside terminals and clinician workstations.

The OLAP side begins when the ETL pipeline extracts data from the OLTP sources, transforms it into analytical schemas (star schema with fact and dimension tables), and loads it into the Data Warehouse. This boundary serves a critical purpose: it isolates heavy analytical queries that could slow down the operational system. A nurse entering a new medication should never experience latency because an executive is running a month-long report. The ETL pipeline runs on a scheduled cadence (hourly for near-real-time dashboards, nightly for comprehensive reporting), ensuring that the warehouse contains a consistent, point-in-time snapshot of the organization without impacting clinical workflows. The OLAP layer supports use cases like trend analysis, cohort studies, resource planning, and regulatory compliance reporting that require scanning millions of records.

## Trade-offs

The most significant architectural trade-off involves the handling of high-frequency ICU vital signs data. Retaining every raw data point from ICU monitors indefinitely would provide the highest fidelity for retrospective clinical research and anomaly detection, but it comes with prohibitive storage costs and diminishing analytical value. A single ICU patient can generate over 100,000 data points per day across multiple monitoring streams, translating to hundreds of gigabytes annually per ICU bed. For a 50-bed ICU, this exceeds 10 TB per year.

To mitigate this, the architecture implements a multi-tier retention and downsampling strategy. Raw second-by-second data is retained in InfluxDB for only 30 days, allowing real-time monitoring and short-term clinical review. After this period, data is automatically downsampled: 1-minute averages for the next 12 months, 5-minute averages for historical years, and 15-minute aggregates for long-term archival. This reduces storage by approximately 95% while preserving clinically meaningful patterns. Critical events like arrhythmias or hypotensive episodes are flagged and stored at full resolution regardless of retention policy, ensuring no clinically significant data is lost. This tiered approach balances the competing demands of storage cost, query performance, and clinical data fidelity.
